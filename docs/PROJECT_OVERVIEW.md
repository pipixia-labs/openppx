# openppx Project Overview

## 1. Positioning

`openppx` is a lightweight agent system built on Google ADK. Its goal is to cover a complete agent runtime path with a compact implementation:

- Multi-channel message integration: local, Feishu, Telegram, WhatsApp, Discord, DingTalk, email, Slack, and QQ
- Skill-driven capability extension through `SKILL.md`
- Built-in tool execution for files, commands, web, messages, cron jobs, and subagents
- Persistent sessions through SQLite and optional long-term memory through ADK Memory Service

It is not just a chat bot. It is an agent runtime foundation that can execute real tasks.

## 2. Core Architecture

### 2.1 Key Modules

- `openppx/agent.py`
  - Defines the root `LlmAgent`.
  - Registers tools, including `PreloadMemoryTool` and `spawn_subagent`.
  - Calls `add_session_to_memory()` from `after_agent_callback` to persist memory.
- `openppx/gateway.py`
  - Runs the gateway loop: consumes inbound messages, calls the ADK Runner, and publishes outbound messages.
  - Handles session commands such as `/help` and `/new`.
- `openppx/runtime/runner_factory.py`
  - Creates `Runner` instances consistently and enables `ResumabilityConfig` plus `EventsCompactionConfig`.
- `openppx/runtime/session_service.py`
  - Provides session storage through SQLite `DatabaseSessionService`.
- `openppx/runtime/memory_service.py`
  - Builds the memory service, either `in_memory` or `markdown`.
- `openppx/runtime/markdown_memory_service.py`
  - Implements local Markdown memory, organized by `app_name` and `user_id`.

### 2.2 Message Processing Flow

1. A channel produces an `InboundMessage`.
2. Gateway parses and routes the message.
3. Gateway builds `UserContent` and calls `runner.run_async(...)`.
4. ADK streams events back, and Gateway merges text output.
5. Gateway publishes an `OutboundMessage` to the target channel.

## 3. Sessions and Memory

### 3.1 Session Model

- User scope: `user_id`
- Conversation scope: `session_id`
- Default session key: `{channel}:{chat_id}`, generated by `InboundMessage.session_key`
- Session persistence: SQLite, defaulting to `~/.openppx/database/sessions.db`

### 3.2 `/new`, `/help`, and `/rewind`

Gateway includes built-in session commands:

- `/help`
  - Returns command help directly.
  - Does not call the model.
- `/new`
  - Attempts to write the active session into memory first, if a memory service is enabled.
  - Binds a new ADK `session_id` to the current `channel:chat_id`.
  - Routes later messages into the new conversation context.
  - Does not call the model.
- `/rewind [last|<invocation_id>]`
  - Uses ADK 2.1 `Runner.rewind_async()` to append a rewind event to the current session.
  - Defaults to rewinding before the latest visible invocation, or accepts an explicit invocation id.
  - Affects only future model-visible ADK session context. It does not undo external side effects such as file writes, messages, command execution, or cron changes.

The current `/new` mapping is in-process. After a process restart, routing falls back to the default `session_key`.

### 3.3 Memory Backends

Select the backend with `OPENPPX_MEMORY_BACKEND`:

- `markdown`, the default
  - Persists local files under `OPENPPX_MEMORY_MARKDOWN_DIR`.
  - Default directory: `~/.openppx/<agent_name>/memory`
- `in_memory`, for debugging
  - Keeps memory inside the current process and does not persist it.

Use `OPENPPX_MEMORY_ENABLED` to enable or disable memory. It is enabled by default.

### 3.4 Markdown Memory Layout

When the backend is `markdown`, the directory layout is:

```text
<memory_root>/
  MEMORY.md
  HISTORY.md
  .event_ids.<app_name>.<user_id>.json
```

- `MEMORY.md`
  - Stores only long-term facts such as preferences, project context, and relationships.
  - Each entry keeps the original conversation timestamp.
  - `search_memory` searches only this file.
- `HISTORY.md`
  - Plain-text conversation transcript, append-only.
- `.event_ids.<app_name>.<user_id>.json`
  - Deduplication index for ingested event ids.

## 4. Context Compaction

`runner_factory` injects `EventsCompactionConfig` into the ADK app.

Configuration:

- `OPENPPX_COMPACTION_ENABLED`, default `1`
- `OPENPPX_COMPACTION_INTERVAL`, default `8`
- `OPENPPX_COMPACTION_OVERLAP`, default `1`
- `OPENPPX_COMPACTION_TOKEN_THRESHOLD`, optional positive integer
- `OPENPPX_COMPACTION_EVENT_RETENTION`, optional non-negative integer

`TOKEN_THRESHOLD` and `EVENT_RETENTION` must be set together. If only one is set, both are ignored to avoid startup errors.

## 5. Tool Capabilities

Built-in tools cover:

- Files: `read_file`, `write_file`, `edit_file`, `list_dir`
- Commands: `exec`
- Web: `web_search`, `web_fetch`
- Messages: `message`, `message_image`
- Scheduling: `cron`
- Async subtasks: `spawn_subagent`
- Skill inspection: `list_skills`, `read_skill`

External toolsets can also be attached dynamically through MCP configuration, using stdio, HTTP, or SSE.

## 6. Configuration and Runtime

### 6.1 Recommended Initialization

```bash
ppx doctor --fix
```

This creates:

- `~/.openppx/<agent_name>/config.json`
- `~/.openppx/<agent_name>/runtime.json`
- Agent metadata files such as `~/.openppx/<agent_name>/AGENTS.md`
- A separate workspace referenced by `agent.workspace`

### 6.1.1 Core Files in the Agent Config Directory

Each agent config directory contains `config.json`, `runtime.json`, and a set of files that define agent behavior, capabilities, and state. The workspace is reserved for task inputs, outputs, code, and temporary artifacts.

#### `IDENTITY.md`

Defines the stable identity of the agent: "who this agent is."

Recommended contents:

- Agent name
- Basic role
- Stable self-description
- Identity attributes that should not change frequently

Keep it short and stable. Do not mix temporary task context into it.

#### `SOUL.md`

Defines values, behavioral tendencies, and communication style: "what principles this agent follows."

Recommended contents:

- Value priorities
- Behavioral style
- Communication preferences
- Principles to follow when requirements conflict

`IDENTITY.md` focuses on who the agent is. `SOUL.md` focuses on how it works.

#### `AGENTS.md`

Defines high-level operating rules and collaboration conventions: "what rules this agent follows while working."

Recommended contents:

- How to proceed after receiving a task
- How to communicate before write operations
- How to use tools
- How to handle uncertainty
- How to collaborate with the user or other agents
- How to use and maintain memory

#### `USER.md`

Defines the user profile and interaction preferences so the agent can adapt to the current user.

Recommended contents:

- User background
- Technical level
- Communication style preferences
- Task-specific constraints
- Preferences about output length and explanation style

#### `TOOLS.md`

Defines the toolset visible to the current agent and the boundaries for tool use.

Recommended contents:

- Available tools
- Purpose of each tool
- Usage limits and safety boundaries
- Recommended or forbidden invocation patterns

`TOOLS.md` describes the action interface layer, not method templates.

#### `skills/`

`skills/` stores reusable skill units.

Each skill is a method template for a class of tasks. It usually includes:

- Applicable scenarios
- Input and output contract
- Recommended steps
- Tool orchestration
- Domain experience and constraints

`skills/` differs from `TOOLS.md`:

- `TOOLS.md` answers "what actions are available."
- `skills/` answers "how to organize those actions for a task."

#### `memory/`

`memory/` stores long-term memory, not every temporary detail from one task.

Typical files:

- `memory/MEMORY.md`
  - Durable, reusable facts such as user preferences, project background, and stable context.
- `memory/HISTORY.md`
  - Raw history or conversation trace, generally append-only.

`MEMORY.md` is distilled long-term memory. `HISTORY.md` is raw historical trace.

#### `HEARTBEAT.md`

`HEARTBEAT.md` defines or records periodic tasks, standing watch items, and runtime heartbeat tasks.

Recommended contents:

- Scheduled checks
- Periodic reminders
- Ongoing watch items
- Status of running periodic tasks

It belongs closer to runtime state than to stable identity.

### 6.2 Common Runtime Modes

```bash
# Single-turn call
python -m openppx.cli -m "Describe what you can do"

# Local gateway mode
python -m openppx.cli gateway run --channels local --interactive-local

# Multi-channel gateway mode
ppx gateway run --channels local,feishu --interactive-local
```

### 6.3 Testing

```bash
source .venv/bin/activate
pytest -q
```

## 8. Fit and Boundaries

Good fits:

- A debuggable, extensible ADK agent foundation that can connect to multiple channels
- Fast iteration on skills and tool pipelines

Current boundaries:

- `/new` session routing is not yet persisted across process restarts.
- Markdown memory uses text append plus keyword search and does not yet include advanced semantic retrieval.
- Channel behavior depends on platform APIs and configuration completeness.

Future focused docs can be added under `docs/`:

- `ARCHITECTURE.md`
- `MEMORY.md`
- `OPERATIONS.md`
- `MCP_INTEGRATION.md`
